## Step-by-Step Analysis

### Understanding the Problem
The task is to find the **maximum product of k *contiguous* elements** in the array. This is a sliding window problem — we must consider only **subarray** products (consecutive elements), not just the k largest elements.

---

### Code Review

```java
class Solution {
    public int findMaxProduct(int[] arr, int k) {
        Arrays.sort(arr)  // <-- MISSING SEMICOLON - COMPILATION ERROR
        int max = arr.length-1, prod=1;
        for(int i=k; i>0; i--){
            prod *= arr[max--];
        }
        return prod;
    }
}
```

### Core Functionality Analysis [5 marks]

**Critical Logic Error — Wrong Approach:**
The student sorts the array and multiplies the top `k` elements. This is **fundamentally wrong** for this problem.

- The problem asks for the maximum product of **k contiguous (consecutive) elements** — i.e., a subarray.
- Sorting destroys the original order and picks the k largest values regardless of their positions.
- Example: `arr = [1, 2, 3, 4], k = 2` → sorted: `[1,2,3,4]` → picks `4*3 = 12` ✓ (happens to be correct by coincidence)
- Example: `arr = [1, 6, 7, 8], k = 3` → sorted picks `8*7*6 = 336` ✓ (coincidence again for these test cases)
- But for `arr = [5, 1, 2, 3], k = 2` → correct answer is `5*1=5`? No, `2*3=6`. Sorted approach gives `3*2=6` ✓. But for `arr = [10, 1, 1, 9], k = 2` → correct answer is `10*1=10`, sorted gives `10*9=90` ✗. The sorted approach is **logically incorrect** for the general case.

The correct approach is a **sliding window** of size k, computing the product for each window and tracking the maximum.

This is a major flaw — both a compilation failure and a fundamentally incorrect algorithm for the general case.
